# D-20260714-04 — Full-repo audits trace semantic implementation

Index: [DecisionHistory.md](../DecisionHistory.md)

## Detail

The audit contract mentioned handlers, persistence, permissions, and tests, but
its mandatory per-file evidence was UI-centered. A non-interface file could
pass with a coverage purpose and generic no-finding note. Explicit placeholder
checks found TODO/stub vocabulary but could not establish whether apparently
finished code fulfilled its actual responsibility.

## Decision

Every batch now records at least one source-backed implementation inventory row
per owned unit and a separate uniquely identified row for each distinct
responsibility in that unit. Each responsibility records an enumerated
authoritative or honestly source-inferred contract basis and parsed or manual
discovery bound to its assigned-unit anchor. Its verification declares test,
runtime, or source-only evidence plus one counterfactual or invariance;
source-only evidence cannot close a persistence, integration, external-effect,
or success PASS. The lead independently reopens and rechecks every batch PASS
anchor—incrementally when needed, but never by sampling—and records the same
typed evidence discipline in its verified reconciliation report.

That report maps every batch Contract ID exactly once into feature and entry-point
traces covering registration, real calculation/domain logic, data lifecycle,
integrations, authorization, failures/recovery, observable outcomes,
operational lifecycle, and verification. Each batch and lead result is derived
from its recorded statuses, a lead trace cannot hide a mapped batch gap or
blocked result, and PASS requires substantive evidence of the real outcome,
side effects, and applicable failure behavior. The review explicitly checks
marker-free substitutes such as hard-coded results, ignored inputs, fake
success/persistence, incomplete plumbing, production mocks, and shallow tests.

Every verified artifact-backed audit creates an external, fully disposed
completion-ledger projection, including an empty projection for a clean audit.
Batch and lead gaps map one-to-one to atomic findings, and a pass-only receipt
binds the verified manifest, report root, and report hashes used for
consolidation. The ledger updater independently reruns verification during both
plan and apply while guarding the verifier's source, report, effort, prompt,
queue/exclusion, and artifact closure, and accepts the receipt only when its
canonical verifier-result digest matches that genuine pass. Repository mutation remains authorization-gated: an explicit
user request or applicable project instruction may authorize the verifier-gated
plan/apply importer, which preserves unrelated active rows, never prunes, and
writes only confirmed unresolved obligations to `CompletionLedger.md`.

## Alternatives and rationale

- Expanding TODO/NotImplemented regexes was rejected as the primary mechanism:
  it cannot understand arbitrary domain intent and broad constant/no-op rules
  would create false positives.
- File hashes plus generic manual notes were rejected because they prove which
  bytes were assigned, not what implementation behavior was traced.
- Directly importing raw worker findings was rejected because hypotheses,
  duplicates, and incomplete evidence do not belong in an authoritative active
  queue.

## Verification

The harness self-test now proves responsibility-level implementation coverage,
rejects missing units or named definitions, duplicate Contract IDs,
missing or invalid basis/discovery provenance, fabricated discovery and lead
anchor propagation, missing counterfactual/invariance or evidence type,
source-only stateful PASS, generic/fabricated/unverified or unbound PASS rows,
trace/result contradictions,
omitted or multiply mapped batch IDs, and lead results that hide mapped gaps.
It exercises marker-free fixed-calculation and cross-file gaps.
The separate `skills/full-repo-audit/evals/marker-free/` suite can measure
manual-agent recall and intentional-lookalike precision for six marker-free gap
classes when fresh agents actually run the cases. Its self-test synthesizes
oracle-derived responses and proves only the evaluation infrastructure; no
empirical agent-performance result is claimed by that self-test. Results from
an actual run would still not be deterministic verifier proof or a guarantee
about future agent judgment.
On 2026-07-14, six isolated fresh-context manual runs were each given only one
case repository and the response schema, not the sibling oracle. All six
expected gaps and all six intentional precision controls matched, with no
unmatched findings (100/100 per case). The runtime did not provide independently
attested model identity, so this remains point-in-time behavioral evidence, not
a portable or permanent recall guarantee.
Ledger tests cover full candidate
disposition, unrelated-row preservation, idempotence, hypothesis exclusion,
symlink rejection, source drift, and exact plan/apply hashes. Standalone skill
validation proves the vendored fallback behaves like the canonical harness.
